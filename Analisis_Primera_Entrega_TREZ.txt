================================================================================
  REQUISITO 1: PATRÓN DE DISEÑO VISITOR
================================================================================

  ESTADO: ✅ CUMPLE

  TREZ implementa el patrón Visitor de forma correcta y completa. Este patrón
  se utiliza como mecanismo principal para recorrer y evaluar el Árbol
  Sintáctico Abstracto (AST) generado por ANTLR4.

  EVIDENCIA EN EL CÓDIGO:

  1. Interfaz base del Visitor (generada por ANTLR4):
     Archivo: src/parser/TrezParserVisitor.py

     Esta clase define la interfaz genérica del Visitor con métodos para
     cada tipo de nodo del AST:

       - visitProgram(ctx)      → Visita el nodo raíz del programa
       - visitLet_stmt(ctx)     → Visita asignaciones con 'let'
       - visitExpr_stmt(ctx)    → Visita sentencias de expresión
       - visitMulDivExpr(ctx)   → Visita nodos de multiplicación/división
       - visitAddSubExpr(ctx)   → Visita nodos de suma/resta
       - visitNumExpr(ctx)      → Visita literales numéricos
       - visitStringExpr(ctx)   → Visita literales de cadena
       - visitVarExpr(ctx)      → Visita referencias a variables
       - visitParenExpr(ctx)    → Visita expresiones con paréntesis
       - visitFuncCallExpr(ctx) → Visita llamadas a funciones
       - visitArrayExpr(ctx)    → Visita expresiones de arreglo
       - visitArray(ctx)        → Visita nodos de arreglo

     Cada método por defecto llama a self.visitChildren(ctx), siguiendo
     fielmente el patrón Visitor clásico del GoF (Gang of Four).

  2. Visitor concreto (implementación propia):
     Archivo: src/visitor.py

     La clase TrezVisitor hereda de TrezParserVisitor (la interfaz base)
     y SOBRESCRIBE cada método visit* con la lógica de evaluación real:

       class TrezVisitor(AntlrTrezVisitor):
           def visitMulDivExpr(self, ctx):
               left = self.visit(ctx.expr(0))
               right = self.visit(ctx.expr(1))
               ...

     En lugar de compilar a bytecode o código máquina, el Visitor recorre
     el AST nodo por nodo ejecutando la operación correspondiente. Esto
     es un uso canónico del patrón Visitor como "Tree-Walking Interpreter".

  3. Punto de entrada del Visitor:
     Archivo: src/main.py

       tree = parser.program()     ← Genera el AST
       visitor = TrezVisitor()     ← Instancia el Visitor concreto
       visitor.visit(tree)         ← Inicia el recorrido del AST

  CONCLUSIÓN:
  El patrón Visitor está implementado correctamente con:
    • Una interfaz base (TrezParserVisitor) con métodos visit* para cada nodo
    • Una implementación concreta (TrezVisitor) que sobrescribe esos métodos
    • El método visit() como dispatcher que selecciona el método adecuado
  Esto es una implementación fiel y funcional del patrón de diseño Visitor.


================================================================================
  REQUISITO 2: PARADIGMA DE PROGRAMACIÓN FUNCIONAL
================================================================================

  ESTADO: ✅ CUMPLE

  TREZ está diseñado bajo un paradigma funcional. El lenguaje prioriza la
  evaluación de expresiones y funciones puras sobre la mutación de estado.

  EVIDENCIA EN EL CÓDIGO:

  1. Evaluación basada en expresiones:
     Todo en TREZ es una expresión que retorna un valor. No hay sentencias
     de control imperativas (for, while, if-else mutable). El programa se
     estructura como una secuencia de expresiones evaluadas funcionalmente.

     Ejemplo de código TREZ:
       (5 + 3) * 2 / 4;     → Evalúa y retorna 4.0
       10 - 2 * 3;           → Evalúa y retorna 4

  2. Inmutabilidad conceptual con 'let':
     La palabra clave 'let' se usa para crear "bindings" (enlaces) de valores
     a nombres, no variables mutables. Esto sigue el estilo de lenguajes
     funcionales como Haskell, OCaml o Elm:

       let x = 10;
       let y = x + 5;

  3. Funciones puras:
     Las funciones built-in del lenguaje (relu, sigmoid, dot, transpose,
     abs, sqrt, pow, exp, log, sin, cos, tan, factorial, etc.) son todas
     funciones puras: reciben entradas y devuelven resultados sin efectos
     secundarios.

  4. Operaciones sobre listas de forma funcional:
     Las operaciones sobre arreglos se implementan de forma funcional
     usando comprensiones de listas (list comprehensions) sin mutar los
     datos originales:

       # En visitMulDivExpr (escalar * arreglo):
       return [left * r for r in right]    ← Crea nueva lista, no muta

       # En visitAddSubExpr (arreglo + arreglo):
       return [l + r for l, r in zip(left, right)]  ← Crea nueva lista

  5. Sin efectos secundarios:
     El README y el Design.md declaran explícitamente:
     "Enfocado en funciones puras, inmutabilidad y evaluación de expresiones
     sin efectos secundarios."

  CONCLUSIÓN:
  TREZ cumple con el paradigma funcional al:
    • Basar toda la ejecución en evaluación de expresiones
    • Usar 'let' para bindings inmutables (no variables mutables)
    • Implementar funciones puras sin side-effects
    • Crear nuevos valores en vez de mutar datos existentes


================================================================================
  REQUISITO 3: GRAMÁTICA TIPO 2 (LIBRE DE CONTEXTO)
================================================================================

  ESTADO: ✅ CUMPLE

  TREZ posee una gramática formal de Tipo 2 (Gramática Libre de Contexto)
  según la jerarquía de Chomsky, definida en archivos ANTLR4 (.g4).

  COMPONENTES DE LA GRAMÁTICA:

  A) ANÁLISIS LÉXICO (Lexer) - Archivo: src/parser/TrezLexer.g4
  ─────────────────────────────────────────────────────────────

     Define los TOKENS (componentes léxicos) del lenguaje:

     CATEGORÍA         │ TOKEN         │ PATRÓN
     ──────────────────┼───────────────┼─────────────────────────────
     Palabras clave    │ LET           │ 'let'
     Símbolos          │ LPAREN        │ '('
                       │ RPAREN        │ ')'
                       │ LBRACK        │ '['
                       │ RBRACK        │ ']'
                       │ COMMA         │ ','
                       │ SEMI          │ ';'
                       │ EQ            │ '='
     Operadores arit.  │ PLUS          │ '+'
                       │ MINUS         │ '-'
                       │ MUL           │ '*'
                       │ DIV           │ '/'
     Literales         │ NUMBER        │ '-'? [0-9]+ ('.' [0-9]+)?
                       │ STRING        │ '"' (~["\r\n\\] | '\\' .)* '"'
     Identificadores   │ ID            │ [a-zA-Z_][a-zA-Z0-9_]*
     Ignorados         │ WS            │ [ \t\r\n]+ -> skip
                       │ LINE_COMMENT  │ '//' ~[\r\n]* -> skip


  B) ANÁLISIS SINTÁCTICO (Parser) - Archivo: src/parser/TrezParser.g4
  ────────────────────────────────────────────────────────────────────

     Define las REGLAS DE PRODUCCIÓN gramaticales (gramática libre de contexto):

     program   → statement+ EOF
     statement → let_stmt | expr_stmt
     let_stmt  → LET ID EQ expr SEMI
     expr_stmt → expr SEMI
     expr      → expr (MUL | DIV) expr          (precedencia alta)
               | expr (PLUS | MINUS) expr       (precedencia baja)
               | NUMBER
               | STRING
               | ID
               | LPAREN expr RPAREN
               | array
               | ID LPAREN (expr (COMMA expr)*)? RPAREN
     array     → LBRACK RBRACK
               | LBRACK expr (COMMA expr)* RBRACK

     Esta gramática es definitivamente de Tipo 2 porque:
       • Todas las producciones tienen la forma A → α, donde A es un
         no-terminal único y α es una cadena de terminales y no-terminales.
       • No depende del contexto que rodea al no-terminal (libre de contexto).
       • Usa recursión (la regla 'expr' es recursiva) lo cual es una
         característica propia de las gramáticas tipo 2 (no posible en tipo 3).

  C) MANEJO DE ERRORES
  ─────────────────────

     El lenguaje tiene un sistema de errores robusto:
     • TrezSyntaxError  → Errores de análisis sintáctico (línea y columna)
     • TrezRuntimeError → Errores de ejecución semántica
     • TrezErrorListener → Listener personalizado de ANTLR4 que captura
                           errores de parsing y los reporta con formato.

  CONCLUSIÓN:
  TREZ posee una gramática formal completa con:
    • Léxico definido formalmente con expresiones regulares (tokens)
    • Sintaxis definida como Gramática Libre de Contexto (Tipo 2 de Chomsky)
    • Separación limpia entre Lexer y Parser
    • Manejo de errores léxicos y sintácticos


================================================================================
  REQUISITO 4: OPERACIONES ARITMÉTICAS (+, -, *, /)
================================================================================

  ESTADO: ✅ CUMPLE

  El lenguaje soporta las cuatro operaciones aritméticas básicas con
  correcta precedencia de operadores y soporte para paréntesis.

  EVIDENCIA EN EL CÓDIGO:

  1. Tokens definidos en el Lexer (TrezLexer.g4):
       PLUS:  '+'
       MINUS: '-'
       MUL:   '*'
       DIV:   '/'

  2. Reglas del Parser con precedencia correcta (TrezParser.g4):
       expr: expr (MUL | DIV) expr       ← Mayor precedencia (primero)
           | expr (PLUS | MINUS) expr    ← Menor precedencia (después)
           | LPAREN expr RPAREN          ← Paréntesis para agrupar

     En ANTLR4, las alternativas listadas primero tienen mayor precedencia,
     así que * y / se evalúan antes que + y -, lo cual es correcto.

  3. Evaluación en el Visitor (visitor.py):

     Multiplicación y División (visitMulDivExpr):
       - Evalúa operandos izquierdo y derecho recursivamente
       - Soporta escalar * arreglo y arreglo * escalar
       - Aplica el operador correspondiente (* o /)

     Suma y Resta (visitAddSubExpr):
       - Evalúa operandos izquierdo y derecho recursivamente
       - Soporta suma/resta componente a componente de arreglos
       - Valida que los arreglos tengan la misma longitud

  4. Ejemplo funcional de TREZ (archivo test_basic.trez):
       (5 + 3) * 2 / 4;    → Resultado: 4.0
       10 - 2 * 3;          → Resultado: 4

  CONCLUSIÓN:
  Las cuatro operaciones aritméticas están implementadas con:
    • Tokens léxicos correctos
    • Precedencia de operadores correcta (* y / antes de + y -)
    • Soporte para paréntesis que alteran la precedencia
    • Evaluación funcional mediante el Visitor
