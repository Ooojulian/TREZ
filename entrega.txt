────────────────────────────────────────────────────────────────────────────────
  REQUISITO 1: PATRÓN DE DISEÑO VISITOR
────────────────────────────────────────────────────────────────────────────────

  ESTADO: ✅ CUMPLE

  TREZ implementa el patrón Visitor de forma correcta y completa. Es el
  mecanismo principal para recorrer y evaluar el AST generado por ANTLR4.

  EVIDENCIA EN EL CÓDIGO:

  1. Interfaz base del Visitor (generada por ANTLR4):
     Archivo: src/parser/TrezParserVisitor.py

     Define la interfaz genérica con métodos para cada nodo del AST:
       - visitProgram(ctx)
       - visitLet_stmt(ctx)
       - visitExpr_stmt(ctx)
       - visitMulDivExpr(ctx)
       - visitAddSubExpr(ctx)
       - visitNumExpr(ctx)
       - visitStringExpr(ctx)
       - visitVarExpr(ctx)
       - visitParenExpr(ctx)
       - visitFuncCallExpr(ctx)
       - visitArrayExpr(ctx)
       - visitArray(ctx)
       (y muchos más — uno por cada regla de la gramática)

     Cada método por defecto llama a self.visitChildren(ctx), patrón GoF.

  2. Visitor concreto:
     Archivo: src/visitor.py — clase TrezVisitor (638+ líneas)

     Hereda de TrezParserVisitor y SOBRESCRIBE cada visitX() con lógica real:

       class TrezVisitor(AntlrTrezVisitor):
           def visitMulDivExpr(self, ctx):
               left  = self.visit(ctx.expr(0))
               right = self.visit(ctx.expr(1))
               ...

     El Visitor recorre el AST nodo a nodo ejecutando la operación
     correspondiente — uso canónico como "Tree-Walking Interpreter".

  3. Punto de entrada:
     Archivo: src/main.py

       tree    = parser.program()     # genera el AST
       visitor = TrezVisitor()        # instancia el Visitor concreto
       visitor.visit(tree)            # inicia el recorrido

  CONCLUSIÓN:
    • Interfaz base (TrezParserVisitor) con métodos visit* para cada nodo
    • Implementación concreta (TrezVisitor) que sobrescribe esos métodos
    • Dispatcher visit() que selecciona el método adecuado por tipo de nodo
  Implementación fiel y funcional del patrón de diseño Visitor (GoF).


────────────────────────────────────────────────────────────────────────────────
  REQUISITO 2: PARADIGMA DE PROGRAMACIÓN FUNCIONAL
────────────────────────────────────────────────────────────────────────────────

  ESTADO: ✅ CUMPLE

  TREZ está diseñado bajo un paradigma funcional estricto.

  EVIDENCIA EN EL CÓDIGO:

  1. Evaluación basada en expresiones:
     Todo en TREZ es una expresión que retorna un valor.

       (5 + 3) * 2 / 4;     → 4.0
       10 - 2 * 3;           → 4

  2. Inmutabilidad con 'let':
     'let' crea bindings de valores a nombres, no variables mutables.
     Sigue el estilo de Haskell, OCaml o Elm:

       let x = 10;
       let y = x + 5;

  3. Funciones puras:
     Todas las funciones de lib/ retornan nuevos valores sin efectos
     secundarios (relu, sigmoid, dot, transpose, mse, etc.).

  4. Operaciones sin mutación:
     Las operaciones sobre arreglos crean nuevas listas:

       return [left * r for r in right]       # escalar * arreglo
       return [l + r for l, r in zip(a, b)]   # arreglo + arreglo

  5. Closures y funciones de primer orden:
     Las funciones capturan el entorno de definición (TrezFunction).
     Las lambdas son valores de primera clase pasables al operador |>.

  CONCLUSIÓN:
    • Ejecución basada en evaluación de expresiones
    • Bindings inmutables con 'let'
    • Funciones puras sin side-effects
    • Nuevos valores en vez de mutación


────────────────────────────────────────────────────────────────────────────────
  REQUISITO 3: GRAMÁTICA TIPO 2 (LIBRE DE CONTEXTO)
────────────────────────────────────────────────────────────────────────────────

  ESTADO: ✅ CUMPLE

  Gramática formal de Tipo 2 (Chomsky) definida en archivos ANTLR4 (.g4).

  A) ANÁLISIS LÉXICO — src/parser/TrezLexer.g4

     CATEGORÍA          │ TOKEN          │ PATRÓN
     ───────────────────┼────────────────┼──────────────────────────────────
     Palabras clave     │ LET, FUNC      │ 'let', 'func'
                        │ RETURN, IF     │ 'return', 'if'
                        │ ELSE, WHILE    │ 'else', 'while'
                        │ FOR, IN, NOT   │ 'for', 'in', 'not'
     Operadores         │ POW            │ '**'
                        │ AND, OR        │ '&&', '||'
                        │ PIPE, ARROW    │ '|>', '->'
                        │ EQEQ, NEQ      │ '==', '!='
                        │ LE, GE         │ '<=', '>='
                        │ MOD            │ '%'
     Símbolos           │ LPAREN...      │ '(', ')', '[', ']', '{', '}'
                        │ COMMA, SEMI    │ ',', ';'
                        │ DOT, COLON     │ '.', ':'
     Literales          │ NUMBER         │ [0-9]+ ('.' [0-9]+)?
                        │ STRING         │ '"' (~["\r\n\\] | '\\' .)* '"'
                        │ BOOL           │ 'true' | 'false'
     Identificadores    │ ID             │ [a-zA-Z_][a-zA-Z0-9_]*
     Ignorados          │ WS             │ [ \t\r\n]+ → skip
                        │ LINE_COMMENT   │ '//' ~[\r\n]* → skip

  B) ANÁLISIS SINTÁCTICO — src/parser/TrezParser.g4

     program    → statement+ EOF
     statement  → let_stmt | bind_tuple | func_def | return_stmt
                | expr_stmt | if_stmt | while_stmt | for_stmt | block

     let_stmt   → LET ID EQ expr SEMI
     bind_tuple → LET LBRACK ID (COMMA ID)* RBRACK EQ expr SEMI
     func_def   → FUNC ID LPAREN param_list? RPAREN block
     for_stmt   → FOR ID IN expr block
     if_stmt    → IF LPAREN expr RPAREN block (ELSE (if_stmt | block))?

     expr (jerarquía de menor a mayor precedencia):
         lambdaDef   → BACKSLASH ID ARROW expr
         pipeOp      → expr PIPE expr
         orExpr      → expr OR expr
         andExpr     → expr AND expr
         notExpr     → NOT expr
         eqExpr      → expr (EQEQ | NEQ) expr
         compareExpr → expr (LT | LE | GT | GE) expr
         addSubExpr  → expr (PLUS | MINUS) expr
         mulDivExpr  → expr (MUL | DIV | MOD) expr
         powExpr     → expr POW expr
         unaryMinus  → MINUS expr
         indexExpr   → expr LBRACK expr RBRACK
         moduleCall  → ID DOT ID LPAREN args? RPAREN
         methodCall  → expr DOT ID LPAREN args? RPAREN
         funcCall    → ID LPAREN args? RPAREN
         // literales, arrays, dicts

     Gramática Tipo 2 porque:
       • Todas las producciones tienen forma A → α (no-terminal único a la izq.)
       • No depende del contexto (libre de contexto)
       • Usa recursión — característica imposible en Tipo 3

  C) MANEJO DE ERRORES

     • TrezSyntaxError      → detectado por TrezErrorListener (ANTLR4)
                              incluye línea, columna y token inesperado
     • TrezRuntimeError     → base para errores de ejecución
     • UndefinedSymbolError → variable no definida (con símbolo y entorno)
     • ShapeMismatchError   → dimensiones incompatibles en tensores
     • MathDomainError      → log(0), sqrt(-1), etc.

  CONCLUSIÓN:
    • Léxico definido con expresiones regulares (tokens)
    • Sintaxis definida como CFG Tipo 2 de Chomsky
    • Separación limpia Lexer / Parser
    • Manejo de errores léxicos y sintácticos


────────────────────────────────────────────────────────────────────────────────
  REQUISITO 4: OPERACIONES ARITMÉTICAS (+, -, *, /)
────────────────────────────────────────────────────────────────────────────────

  ESTADO: ✅ CUMPLE

  EVIDENCIA EN EL CÓDIGO:

  1. Tokens: PLUS '+', MINUS '-', MUL '*', DIV '/', MOD '%', POW '**'

  2. Reglas con precedencia correcta (TrezParser.g4):
       mulDivExpr → expr (MUL | DIV | MOD) expr   ← mayor prec.
       addSubExpr → expr (PLUS | MINUS) expr       ← menor prec.
       powExpr    → expr POW expr                  ← mayor que mul/div

  3. Evaluación en visitor.py:
       visitMulDivExpr → soporta escalar*arreglo y arreglo*escalar
       visitAddSubExpr → soporta suma componente a componente de arreglos
       visitPowExpr    → potenciación con precedencia derecha

  4. Verificado en tests:
       (5 + 3) * 2 / 4;  → 4.0
       10 - 2 * 3;        → 4
       2 ** 8;            → 256.0

  REQUISITO 1: OPERADOR PIPE |>
────────────────────────────────────────────────────────────────────────────────

  ESTADO: ✅ CUMPLE

  `A |> f` equivale a `f(A)`. Tiene la menor precedencia del lenguaje.
  Permite encadenar transformaciones declarativas:

    let resultado = [1, 2, 3] |> relu |> sigmoid;
    datos |> \x -> Tensordoz.dot(W, x) |> Mathdoz.relu;

  EVIDENCIA:
    • Tokens PIPE '|>' en TrezLexer.g4
    • Regla pipeOp en TrezParser.g4 (menor precedencia)
    • visitPipeOp en visitor.py: evalúa lado izquierdo, lo pasa al derecho

  REQUISITO: LECTURA DE DATOS REALES (CSV / XLSX)
────────────────────────────────────────────────────────────────────────────────

  ESTADO: ✅ CUMPLE

  Datadoz absorbe la lectura de archivos de datos estructurados.
  Ya no se necesita IOdoz.leer() para datos tabulares.

  NUEVAS FUNCIONES EN Datadoz (src/lib/datadoz/datadoz.py):

    read_csv(ruta, delimitador):
      Lee un CSV y retorna lista de dicts {columna: valor}.
      Infiere tipos automáticamente: int, float o str.
      Delimitador configurable (por defecto ',').

    read_xlsx(ruta):
      Lee un .xlsx y retorna lista de dicts {columna: valor}.
      La primera fila es la cabecera.
      Requiere openpyxl (instalación: pip install openpyxl).

    columna(datos, nombre):   extrae columna como lista de valores
    fila(datos, idx):         retorna fila como dict
    num_filas(datos):         cantidad de registros
    num_columnas(datos):      cantidad de columnas
    columnas(datos):          lista de nombres de columnas

  EVIDENCIA EN TREZ (tests/features/test_xlsx.trez):

    let datos = Datadoz.read_xlsx("../tests/data.xlsx");
    mostrar(Datadoz.num_filas(datos));          // 10280
    mostrar(Datadoz.columnas(datos));           // [id, edad, nivel_edu, ...]
    let edades = Datadoz.columna(datos, "edad");
    let primera = Datadoz.fila(datos, 0);
